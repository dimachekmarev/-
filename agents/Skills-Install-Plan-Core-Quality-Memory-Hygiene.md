## Цель
Поднять качество ответов и убрать дрейф/мусор в памяти за счёт:
1) усиления quality gate (контроль качества перед финальным ответом),
2) управляемой памяти (что хранить/не хранить),
3) безопасной установки только проверенных skills.

## Пакет B — Content Ops / Memory Hygiene (правило применения)

Эти навыки используем как поведенческий слой:
- перед публикацией: короткая проверка уникальности контента/изображений;
- перед сохранением в memory: фильтр Keep/Delete/Rewrite/Limit;
- при конфликте памяти и файлов: приоритет у source of truth (Obsidian/регламент).

Рекомендуемый runtime-чек (перед финальным ответом):
1. Это факт или предположение?
2. Есть источник (файл/лог/ссылка)?
3. Не устарело ли это?
4. Нужно ли это именно в memory, а не в wiki/skills?

## Блокировано сканером безопасности (не установлено)

### DANGEROUS (автоблок, override невозможен)
- `pr-review-ci-fix`
- `codebase-migrate`
- `sentry-triage`
- `issue-triage`

Причина: обнаружен supply-chain паттерн `curl ... | bash` внутри SKILL.md.

### CAUTION (можно только после ручной ревизии)
- `skill-creator`

Причина: флаг на потенциальный эксфильтрационный паттерн в инструкциях.

---

## Как продолжать безопасно

1. Оставить установленные 10 навыков как «боевой минимум».
2. Для каждого block/caution навыка:
   - скачать SKILL.md,
   - вычистить опасные команды,
   - переустановить из локального проверенного файла.
3. После этого запустить `hermes skills list` и контрольный smoke-test (короткая задача на каждый новый навык).

---

## Примечание по эксплуатации

Если новый skill начинает тянуть устаревшие/опасные практики:
- сразу патчить (`skill_manage(action='patch')`) или удалять,
- фиксировать в отдельном «контракте памяти» запрет на перенос непроверенных инструкций из внешних текстов в durable memory.
