# Phase 1 Money Room — 48h delivery checklist

Offer: **AI-бот для приёма заявок за 48 часов**.

Status: **delivery checklist only**. Это операционный чеклист для проверки 48h-обещания. Сборки, деплой, подключение клиентов, рассылки и любые внешние изменения из этой задачи **не запускались**.

## 0. Главный принцип delivery

48 часов считаются **только после**:
1. оплаты или явного approval на старт;
2. заполненного intake;
3. безопасной передачи доступов без секретов в открытом виде;
4. подтверждённого канала, куда будет приходить заявка;
5. QA go/no-go по обещаниям и рискам.

Если хотя бы один пункт отсутствует — клиенту нельзя обещать точный запуск за 48 часов. Формулировка меняется на: `после материалов и доступов подготовим безопасный MVP-план запуска`.

## 1. Минимальный deliverable за 48 часов

### Что реально можно обещать в MVP

- 1 входной канал: Telegram / VK / сайт-форма / WhatsApp через безопасный промежуточный сценарий, если доступен.
- 1–3 услуги в scope.
- 3–7 уточняющих вопросов для заявки.
- Ответы только по утверждённому FAQ.
- Сбор контакта: имя + телефон или мессенджер + услуга + удобное время + комментарий.
- Handoff менеджеру: Telegram-сообщение, таблица или CRM-ready файл.
- Ручная проверка первых тестовых заявок.
- Инструкция для менеджера: что делать после получения заявки.

### Что НЕ обещать в 48h без отдельной оценки

- Закрытая CRM-интеграция, если нет готового API/доступа/вебхука.
- Платежи внутри бота.
- Полный omnichannel (все каналы сразу).
- Юридические, медицинские, финансовые консультации.
- Самостоятельное назначение скидок, запись без подтверждения менеджера, обещание результата.
- Хранение паспортных данных, медицинских данных, карт, банковских реквизитов.
- Продакшн-деплой в клиентскую инфраструктуру без отдельной команды и approval.

## 2. Intake перед стартом

Заполняется до обещания точного срока.

| Блок | Вопрос | Минимум для старта | Статус |
|---|---|---|---|
| Клиент | Название, ниша, контакт владельца/менеджера | есть ответственный человек | pending |
| Услуги | Какие 1–3 услуги бот принимает | список услуг и короткое описание | pending |
| Канал входа | Где клиент пишет сейчас | выбран один канал для MVP | pending |
| Частые вопросы | Что можно отвечать автоматически | 5–10 утверждённых FAQ | pending |
| Запретные темы | На что бот не отвечает | список стоп-тем | pending |
| Handoff | Куда приходит заявка | Telegram / таблица / CRM-ready формат | pending |
| Менеджер | Кто отвечает после заявки | имя/роль + рабочие часы | pending |
| Успех 7 дней | Что считать полезным тестом | 1 измеримый критерий | pending |
| Approval | Кто утверждает текст бота | владелец/админ/Дмитрий | pending |

### Copy-paste intake для клиента

```text
Чтобы безопасно запустить бот-заявку в срок до 48 часов, нужны материалы:

1) Какие 1–3 услуги принимаем через бота?
2) Где клиент будет писать: Telegram, VK, сайт-форма, WhatsApp или другой канал?
3) Какие 3–7 вопросов бот должен задать перед передачей заявки?
4) Какие 5–10 частых вопросов бот может отвечать сам?
5) На какие темы бот НЕ должен отвечать и должен передать человеку?
6) Куда отправлять готовую заявку: Telegram, таблица или CRM-ready файл?
7) Кто менеджер после заявки и в какие часы он отвечает?
8) Что считаем успехом теста за первые 7 дней?

Важно: пароли, токены, банковские реквизиты и приватные данные не отправляйте в чат. Для доступов используем безопасный способ передачи.
```

## 3. Доступы без секретов в открытом виде

Правило: **не просить и не хранить секреты в обычном чате, Obsidian, CSV, КП или комментариях Kanban**.

| Тип доступа | Безопасный вариант | Что нельзя делать |
|---|---|---|
| Telegram bot token | клиент создаёт bot через BotFather и передаёт токен через 1Password/секрет-хранилище/одноразовую ссылку | вставлять токен в Markdown, Telegram, CRM или issue |
| VK/WhatsApp/CRM API | отдельный технический контакт + временный ключ с минимальными правами | просить основной пароль владельца |
| Сайт/хостинг | staging-доступ или отдельный deploy user | использовать личный root/админ-пароль клиента |
| Таблица | отдельная таблица/папка с доступом только на проект | открывать весь Google/Yandex Drive |
| Уведомления менеджеру | chat id / webhook без лишних прав | давать агенту доступ к личной переписке |
| FAQ/контент | текстовый файл или форма intake | брать ответы из приватных переписок без разрешения |

### Безопасная формулировка клиенту

```text
Секреты и пароли не присылайте обычным сообщением.
Если понадобится токен/ключ, дадим точный список полей и безопасный способ передачи: одноразовая ссылка, 1Password/Bitwarden, временный ключ с минимальными правами или отдельный технический аккаунт.
```

## 4. 48h execution plan

### T+0 — старт после оплаты и материалов

Цель: зафиксировать scope и не дать ложное обещание.

- [ ] Проверить, что оплата/approval на старт есть.
- [ ] Проверить intake: услуги, канал, FAQ, handoff, ответственный менеджер.
- [ ] Проверить, что нет запроса на сложную CRM/платежи/закрытую интеграцию в 48h MVP.
- [ ] Зафиксировать `Client scope handoff` по шаблону ниже.
- [ ] Назначить владельцев: delivery, QA, клиентский менеджер.
- [ ] Подтвердить safe access plan (без секретов в открытом виде).
- [ ] Сформулировать go/no-go: можно ли обещать 48h.

Exit criteria T+0:
- scope узкий;
- материалы есть;
- канал выбран;
- риски записаны;
- сборка/деплой не запускаются без отдельной команды.

### T+6 — сценарий и первый рабочий контур

Цель: готовый сценарий заявки и безопасный handoff.

- [ ] Описать сценарий: приветствие → 3–7 вопросов → итоговая карточка → передача менеджеру.
- [ ] Подготовить FAQ-ответы только из утверждённых материалов.
- [ ] Добавить fallback: `передам вопрос человеку`.
- [ ] Подготовить формат карточки заявки.
- [ ] Подготовить тестовые заявки: минимум 5 happy path и 5 edge cases.
- [ ] QA проверяет тон, запретные темы, отсутствие гарантий продаж.
- [ ] Если нужен токен/ключ — запросить безопасно, не в открытом виде.

Exit criteria T+6:
- сценарий читается человеком;
- нет спорных консультаций;
- понятно, куда уходит заявка;
- есть тест-кейсы.

### T+24 — MVP собран на staging / safe sandbox

Цель: проверить работу без риска для клиента.

- [ ] Собрать MVP только после отдельной команды на build/setup.
- [ ] Использовать staging/sandbox, если доступен.
- [ ] Прогнать тестовые заявки.
- [ ] Проверить, что менеджер получает понятную карточку.
- [ ] Проверить отказные сценарии: непонятный вопрос, запретная тема, агрессивный клиент, просьба о скидке.
- [ ] Проверить privacy: не собираем лишние персональные данные.
- [ ] Записать найденные дефекты и решения.

Exit criteria T+24:
- 10/10 базовых тестов пройдены или дефекты имеют обходной путь;
- менеджер понимает handoff;
- QA не видит блокирующих рисков.

### T+48 — controlled launch / handoff

Цель: передать клиенту работающий MVP или честно зафиксировать блокер.

- [ ] Получить final QA go/no-go.
- [ ] Получить client approval на тексты и handoff.
- [ ] Запустить только после явной команды на deployment/activation.
- [ ] Провести 1–3 контрольные заявки вместе с менеджером.
- [ ] Передать короткую инструкцию менеджеру.
- [ ] Зафиксировать, что клиент считает успехом первых 7 дней.
- [ ] Назначить окно наблюдения: первые 24–72 часа.
- [ ] Обновить KPI/revenue-controller: запущено / blocked / needs changes.

Exit criteria T+48:
- бот принимает заявку в согласованном канале;
- менеджер получает карточку;
- клиент знает ограничения;
- следующий шаг записан.

## 6. Risk register

| Риск | Когда возникает | Красный флаг | Решение | Owner |
|---|---|---|---|---|
| Scope шире 48h | клиент просит CRM, платежи, много каналов | нельзя объяснить MVP одной фразой | урезать до 1 канала / 1–3 услуг | delivery-manager |
| Нет материалов | клиент не дал FAQ/услуги | бот должен «сам придумать ответы» | не стартовать 48h clock | client owner |
| Секреты в чате | прислали токен/пароль открыто | token/password в Telegram/MD | остановить, удалить/заменить ключ, перевыпустить безопасно | delivery + client |
| Юр/мед/фин консультации | клиника/финансы/право | бот советует лечение/деньги/право | только FAQ + передача человеку | QA |
| Handoff не определён | непонятно, кто отвечает | заявки уходят в никуда | назначить менеджера до запуска | client owner |
| Канал технически закрыт | нет API/вебхука/доступа | нужен root/admin без проверки | staging/ручной MVP или убрать канал | delivery |
| Ложные обещания продаж | клиент ждёт гарантию оплат | «увеличим продажи на X» | заменить на «ускорим приём заявок» | proposal + QA |
| Приватные данные | клиент хочет собирать лишнее | паспорт/карта/медкарта | не собирать в MVP | QA |
| Нет QA окна | запуск без тестов | 0 тестовых заявок | блокировать launch до теста | QA |

## 7. Go / no-go перед обещанием 48h

### Можно обещать 48h, если:

- [ ] один канал входа;
- [ ] 1–3 услуги;
- [ ] FAQ и запретные темы утверждены;
- [ ] handoff менеджеру понятен;
- [ ] не нужны закрытые CRM/платежи/сложный календарь;
- [ ] доступы передаются безопасно;
- [ ] QA готов проверить до запуска;
- [ ] есть явная команда перед build/deploy.

### Нельзя обещать 48h, если:

- [ ] клиент не дал материалы;
- [ ] просит несколько каналов сразу;
- [ ] требуется закрытая интеграция без доступов и API;
- [ ] бот должен консультировать по спорным темам;
- [ ] нет ответственного менеджера;
- [ ] секреты передаются в открытом виде;
- [ ] нет approval на сборку/деплой.
